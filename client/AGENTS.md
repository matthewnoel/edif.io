# client Agent Guide

- SvelteKit frontend for all game variants; connects via websocket.
- Client-only blob simulation for visual movement.

## Key Files

- `src/routes/+page.svelte`: lobby.
- `src/routes/room/[code]/+page.svelte`: in-game UI (includes power-up rings/toasts/effects).
- `src/lib/game/connection.svelte.ts`: reactive WebSocket state (includes power-up offer handling).
- `src/lib/game/protocol.ts`: message types (keep aligned with `core/src/protocol.rs`; `protocol.contract.test.ts` pins the JSON shapes on both sides).
- `e2e/home.test.ts`: Playwright smoke tests of the home page; run in CI against the preview build.
- `e2e-fullstack/roundtrip.test.ts`: full-stack Playwright test that boots the Rust server + Vite dev (via `playwright.fullstack.config.ts`) and walks Create Room → Start Match → submit one correct answer.
- `src/lib/game/sim.ts`: clumping/orbit simulation.
- `src/lib/components/PowerUpBadge.svelte`: power-up badge component.

For power-up details see `.cursor/skills/powerups/SKILL.md`.

## Rules

- No gameplay authority on client (sizes, winners, room state are server-driven).
- Keep styles minimal; keep debug panel lightweight.

## Internationalization

Two independent localization axes (a language on one need not exist on the other):

- **Viewer language** (per-player, frontend): all UI plus adapter "chrome" (mode/option/choice labels, input placeholders). Powered by Paraglide JS. Call messages via `import { m } from '$lib/paraglide/messages'`. Adapter chrome originates on the backend in English and is localized client-side via `src/lib/i18n/chrome.ts`, keyed by stable ids (`gameKey`/option `key`/choice `value`), falling back to the server's English label for unknown keys.
- **Content language** (per-mode, backend): e.g. the keyboarding "Word Set" option. Selected by the host, ridden through `gameOptions` — no protocol change.

Catalogs live in `messages/`. `en.json` is the hand-authored source of truth; **`l33t.json` is generated** by `npm run leetify` (`scripts/leetify.mjs` + the pure transform in `src/lib/i18n/leetify.js`) — do not hand-edit it. The leetify transform preserves `{placeholders}` and `**bold**` markers. `npm run leetify:check` (CI) fails on drift; `make generate` regenerates it.

**To add a language, follow `.cursor/skills/i18n/SKILL.md`** — it has the full step-by-step recipe for both a viewer language and a keyboarding content word set. In short: add the locale to `project.inlang/settings.json`, create `messages/<locale>.json` with the same keys as `en.json` (including its endonym `lang_name_*`), then `npm run leetify`. No Svelte/TS edits are needed — `LanguageSwitcher.svelte` derives names dynamically from `lang_name_<locale>` and `chrome.ts` localizes adapter chrome from whatever keys exist. `coverage.test.ts` auto-checks key parity for every catalog.

Paraglide compiles to `src/lib/paraglide/` (gitignored; regenerated by `npm run paraglide:compile`, wired into `prepare`/`check`/`lint`/`test:unit` and the Vite build). The message-format plugin is loaded from `node_modules` (not a CDN) so builds are hermetic. Locale strategy is `['localStorage', 'baseLocale']`: SSR/first paint is always English and the client re-renders after the user picks a language (`LanguageSwitcher.svelte`, which reloads) — do not "fix" this by adding URL routing.
